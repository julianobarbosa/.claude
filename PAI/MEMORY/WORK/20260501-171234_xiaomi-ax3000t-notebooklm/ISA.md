## Problem

Juliano wants curated knowledge about the Xiaomi AX3000T plus actionable best-practice configuration for running 3 of these in a single home, but no consolidated source of truth exists. Vendor docs are sparse and English-incomplete, the OpenWrt/Padavan/ImmortalWrt landscape is fragmented, the strongest practical wisdom lives in dispersed Reddit/Quora/Discord threads, and there is no single live artifact that lets him explore that knowledge interactively or generate audio/mind-map briefings on demand. The AX3000T is also routinely confused with the Wi-Fi 7 BE3000/BE5000 successors, and that name collision pollutes most search results.

## Vision

A NotebookLM notebook titled `xiaomi :: ax3000t :: docs` populated with high-signal sources spanning hardware specs, firmware options, security caveats, and 3-unit single-home deployment patterns drawn from authoritative + community channels — with Studio outputs (Audio Overview + Mind Map, optional Video Overview) so Juliano can listen to a 10-minute briefing while commuting, scan the mind map to choose between Mesh / AP-mode-with-wired-backhaul / AP-mode-with-wireless-backhaul, and ask follow-up questions grounded in cited sources. Convergence feels like: "I know exactly how to deploy these three boxes optimally, every claim is traceable, and the notebook keeps answering questions I haven't thought to ask yet."

## Out of Scope

- Replacing the routers with a different brand (the user owns the AX3000Ts).
- Custom-firmware flashing tutorials beyond a one-paragraph comparison + risk note (OpenWrt wiki is canonical).
- Carrier-grade or enterprise multi-controller setups (UniFi/Omada/etc.).
- ISP-side configuration (PPPoE specifics, IPv6 transition tunnels, CGNAT escape hatches) beyond a single best-practice paragraph.
- Hand-coding a local markdown-only deliverable in lieu of NotebookLM (option 2 was declined).
- Generating Studio outputs that NotebookLM does not natively produce.
- Long-form how-to-flash-OpenWrt step-by-step beyond linking the canonical wiki entry.
- Vendor-marketing narrative without source citations.

## Principles

- **Sources beat assertions.** Every claim must trace to a citation in the notebook. Hard-to-vary explanations (Deutsch) — every recommendation should fail under a specific contrary fact, never re-explain it.
- **Community wisdom is signal, not noise.** Reddit/Quora/Discord debates capture real-world failure modes vendor docs hide; ignoring them produces brittle configs.
- **Three radios in one home is an RF problem before it is a software problem.** Physics first — channel plan, separation, Tx power — software is downstream of geometry.
- **Wired backhaul beats wireless backhaul whenever feasible, full stop.** This is doctrine, not a preference.
- **Operator-first.** The deliverable must be usable by Juliano without re-reading everything; that is what Studio outputs are for.
- **Reach (Deutsch).** Recommendations should generalize beyond this exact home — adding a fourth unit later should not require re-deriving the topology.
